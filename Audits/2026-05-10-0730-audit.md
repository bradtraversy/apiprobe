# Code Audit — 2026-05-10 07:30

**Target:** `.` · **Stack:** Next.js 15.5 + React 19 + TypeScript (client-only, no API routes) · **Files scanned:** 35

## Summary

API Probe is a client-side-only API testing tool — no server, no API routes, no secrets, no auth. The security posture for *itself* is fine (no live exploit found, no leaked credentials), but two security primitives advertised in the README are doing harm: the **DOMPurify-on-every-response** sanitizer silently mutates response bodies (a correctness bug for an API tester), and the **header-key input** in two places loses focus on every keystroke, breaking basic UX. The "rate limit" and a few input validators are also weaker than the README implies. Fix the two focus-loss bugs and the response-mutation bug first; the rest is cleanup.

## High (3)

### 1. Response viewer silently strips HTML/XML tags from response bodies
- **File:** [src/components/api/response-viewer.tsx:71-79, 119-120, 133](src/components/api/response-viewer.tsx#L71-L79)
- **Category:** Correctness · Misuse of sanitization
- **What:** `sanitizeResponse` calls `DOMPurify.sanitize(body, { ALLOWED_TAGS: [], ALLOWED_ATTR: [], KEEP_CONTENT: true })` on every header key/value and on the response body. With `ALLOWED_TAGS: []` + `KEEP_CONTENT: true`, all tags are stripped but inner text is kept. So a real API response of `<html><body>Hello</body></html>` is displayed to the user as `Hello`, and `{"x":"<script>foo</script>"}` becomes `{"x":"foo"}`.
- **Why it matters:** This is an API testing tool — the headline feature is "show me what the API actually returned." Quietly mutating the body defeats the product. Worse, the README ([README.md:51](README.md#L51)) explicitly markets this behavior as XSS protection, but React already escapes any string rendered as a text node, so DOMPurify here is **both unnecessary and harmful**.
- **Fix:** Remove `sanitizeResponse` entirely. Render `key`, `value`, and the formatted body directly inside `<span>` / `<pre>` — React's text-node escaping already prevents any HTML/script execution. Drop the `dompurify` and `@types/dompurify` deps.

### 2. Header-name input loses focus on every keystroke
- **File:** [src/components/api/headers-editor.tsx:53-65](src/components/api/headers-editor.tsx#L53-L65)
- **Category:** Clean code · UX-breaking React anti-pattern
- **What:** The list iterates `Object.entries(headers)` and uses the header name itself as the React `key={key}`. The Input's `onChange` deletes the old key and inserts a new one (`delete newHeaders[key]; newHeaders[e.target.value] = value`), which changes the React key on the next render. React unmounts the old `<div>` + `<Input>` and mounts a fresh one, so the input loses focus after every single character. Editing a header name is effectively impossible — you can only type one character at a time before re-clicking the field.
- **Why it matters:** Headers editing is a core workflow. This breaks it.
- **Fix:** Stop using the header name as the React key. Track headers as an array of `{id: string, key: string, value: string}` (id from `crypto.randomUUID()` once at insert time), use `key={id}` for the row, and only flatten back to `Record<string, string>` when sending the request. Same pattern used in `url-encoded-editor.tsx` works fine.

### 3. Environment variable name input loses focus on every keystroke
- **File:** [src/components/ui/environment-manager.tsx:268-286](src/components/ui/environment-manager.tsx#L268-L286)
- **Category:** Clean code · UX-breaking React anti-pattern
- **What:** Same bug as #2 inside the environment editor modal. `<div key={key}>` over `Object.entries(editingEnvironment.variables)`, with the rename handler deleting + re-inserting the key in the variables object. Renaming a variable forces re-mount per keystroke, so users can't actually rename existing variables — they have to delete and re-add.
- **Why it matters:** Same as #2 — environments are advertised as a flagship feature.
- **Fix:** Same as #2 — store variables as an array of `{id, key, value}` while editing, flatten to `Record<string, string>` on save.

## Medium (9)

### 4. DOMPurify runs on every render, even on huge response bodies
- **File:** [src/components/api/response-viewer.tsx:117-120, 133](src/components/api/response-viewer.tsx#L117-L133)
- **Category:** Optimization
- **What:** `sanitizeResponse` is invoked inline in JSX for every header pair and once for the entire body, with no `useMemo`. Every parent re-render (e.g. when rate-limit ticks every 1s — see [src/components/ui/rate-limit-settings.tsx:17](src/components/ui/rate-limit-settings.tsx#L17), which lives in the sticky header and triggers re-renders that bubble through the page) re-parses the entire response body through DOMPurify.
- **Why it matters:** A 5 MB JSON response means re-running DOMPurify ~once per second. UI jank guaranteed.
- **Fix:** Per finding #1 the right move is removing DOMPurify entirely. If kept for some reason, wrap the call in `useMemo` keyed on `response.body`/`response.headers`.

### 5. Rate limit is purely client-side and trivially bypassable
- **File:** [src/lib/rate-limit.ts:1-74](src/lib/rate-limit.ts#L1-L74), [README.md:48-55](README.md#L48-L55)
- **Category:** Security · Misleading guarantee
- **What:** `requestCount`, `windowStart`, and the `pThrottle` instance are module-level variables in the user's own browser. A page reload or DevTools edit resets the counter; another tab / incognito / curl bypasses it entirely. The fetch goes from the user's browser straight to the target API — there's no server-side bottleneck.
- **Why it matters:** README ([README.md:48-50](README.md#L48-L50)) lists "Rate Limiting: 5 requests per minute to prevent API abuse" under **Security Features**. It is not a security feature — it cannot prevent abuse, only nudge a cooperative user.
- **Fix:** Either (a) reword README to "client-side throttle for accidental burst protection" / drop from the security section, or (b) implement true rate limiting via a Next.js API route proxy (which would also enable real SSRF mitigation and server-side telemetry).

### 6. Header-name regex rejects valid RFC 7230 header names
- **File:** [src/lib/validation.ts:69](src/lib/validation.ts#L69)
- **Category:** Correctness
- **What:** `sanitizeHeaders` only accepts `^[a-zA-Z0-9\-_]+$` for header names. Real HTTP allows `! # $ % & ' * + . ^ _ ` | ~` per RFC 7230 §3.2.6 — so legitimate headers like `X-Trace.Id`, `X-Request.ID`, or any custom dotted/period-bearing header are silently dropped before being sent.
- **Why it matters:** Silent drop, no user feedback. A user testing an API that needs an unusual but valid header will see "their header didn't make it" with no clue why.
- **Fix:** Use the RFC 7230 token charset `^[!#$%&'*+\-.^_`|~0-9A-Za-z]+$`, or just drop the regex check (the browser's `fetch` already rejects invalid header names with a clear error).

### 7. `isSafeInput` over-blocks legitimate header / env-var values
- **File:** [src/lib/validation.ts:39-54](src/lib/validation.ts#L39-L54)
- **Category:** Correctness · False-positive validation
- **What:** Header/env-var values are rejected if they match `/data:/gi`, `/javascript:/gi`, or `/on\w+\s*=/gi` (anywhere). That blocks legitimate values: `Content-Disposition: attachment; filename="ondata=2026.json"` (matches `on\w+\s*=`), env var values like `data:image/png;base64,...` (a real use for an API testing tool), or any `<input>` HTML body containing `onclick=`. The values are sent over `fetch` and never rendered as HTML, so XSS isn't a concern in this code path.
- **Why it matters:** Users will hit "Invalid variable value. Contains potentially dangerous content." for legitimate inputs and have no way to bypass it.
- **Fix:** Drop `isSafeInput` from the header/env-var validation path entirely. The values aren't rendered as HTML anywhere — there's no XSS surface to defend.

### 8. No fetch timeout — slow / hanging APIs lock the UI
- **File:** [src/lib/api-service.ts:28-32](src/lib/api-service.ts#L28-L32)
- **Category:** Optimization · Resilience
- **What:** `fetch(request.url, ...)` has no `signal: AbortSignal.timeout(...)` and no manual `AbortController`. A request to a slow / unreachable API never resolves. The send button stays disabled (`isLoading=true` in [request-form.tsx:64](src/components/api/request-form.tsx#L64)) until the user reloads the page.
- **Why it matters:** Common when testing flaky APIs — the whole tool wedges.
- **Fix:** Add `signal: AbortSignal.timeout(30_000)` (or surface a timeout in settings) and treat the resulting `AbortError` as a user-visible "Request timed out" toast.

### 9. No response-size cap — multi-MB bodies stall the browser
- **File:** [src/lib/api-service.ts:35](src/lib/api-service.ts#L35)
- **Category:** Optimization · Resilience
- **What:** `await response.text()` reads the whole body unconditionally, then it gets stored in React state, then re-rendered + re-DOMPurify'd on every parent render. Hitting an endpoint that returns 50 MB of JSON crashes the tab.
- **Why it matters:** Public deployment at apiprobe.dev — easy to wedge for an unfamiliar user testing a wrong endpoint.
- **Fix:** Stream via `response.body.getReader()` with a hard byte cap (e.g. 5 MB), and show a "Response truncated at N MB" notice when exceeded.

### 10. CSP `script-src` includes `'unsafe-eval' 'unsafe-inline'`
- **File:** [next.config.ts:32](next.config.ts#L32)
- **Category:** Security
- **What:** `script-src 'self' 'unsafe-eval' 'unsafe-inline'` defeats most of CSP's XSS mitigation — any injected `<script>` or `eval(...)` would run.
- **Why it matters:** Next 15 with the App Router can usually run with strict-dynamic + per-request nonces in production. `unsafe-eval` is sometimes needed by dev-mode HMR but shouldn't be in prod CSP.
- **Fix:** Use Next.js's nonce-based CSP (`<NextScript nonce>` pattern via middleware), drop `unsafe-eval` and `unsafe-inline` for production builds. If dev mode breaks, gate the relaxed values on `process.env.NODE_ENV !== 'production'`.

### 11. `images.domains` is deprecated in Next 15 (use `remotePatterns`)
- **File:** [next.config.ts:80-84](next.config.ts#L80-L84)
- **Category:** Clean code · Deprecation
- **What:** `images: { domains: [], ... }`. Next.js deprecated `domains` in favor of `remotePatterns` from 12.3+; in Next 15 this still works but emits a config warning.
- **Why it matters:** Will start failing builds in Next 16. Currently no `<Image>` usage in the repo so this config does nothing — could be dropped entirely.
- **Fix:** Remove the `images` block, or replace with `images: { remotePatterns: [] }`.

### 12. `X-XSS-Protection` header is deprecated and counter-productive
- **File:** [next.config.ts:9-11](next.config.ts#L9-L11)
- **Category:** Security · Outdated guidance
- **What:** Modern browsers (Chrome 78+, Firefox, Edge) have removed the XSS Auditor that this header controlled. Current MDN/OWASP guidance is `X-XSS-Protection: 0` because the legacy auditor introduced its own vulnerabilities.
- **Why it matters:** Cosmetic, but the README ([README.md:49](README.md#L49), § Security Features) lists this as a hardening control — it isn't.
- **Fix:** Set value to `0`, or remove the header.

### 16. `formatBody` re-parses JSON on every render
- **File:** [src/components/api/response-viewer.tsx:60-69, 133](src/components/api/response-viewer.tsx#L60-L69)
- **Category:** Optimization
- **What:** `JSON.parse(body)` + `JSON.stringify(...)` runs every render of the viewer. For a large JSON body this is wasted CPU.
- **Fix:** `useMemo(() => formatBody(response.body, contentType), [response.body, contentType])`.

### 17. AuthEditor `useEffect` depends on parent-supplied object
- **File:** [src/components/api/auth-editor.tsx:19-36](src/components/api/auth-editor.tsx#L19-L36)
- **Category:** Clean code · React subtlety
- **What:** `useEffect(..., [headers])` — `headers` is a fresh object reference whenever the parent setState fires. The effect re-derives `token` from the auth header every render, which can clobber in-flight user typing if the parent re-renders for an unrelated reason.
- **Fix:** Either lift auth state up, or compare the actual auth-header string instead of the object reference (`useEffect(..., [headers['Authorization'] ?? headers['authorization']])`).

### 18. `responseText.length` is character count, not byte count
- **File:** [src/lib/api-service.ts:51](src/lib/api-service.ts#L51)
- **Category:** Correctness · Reporting accuracy
- **What:** The "size" displayed in the viewer is JS string length (UTF-16 code units), not bytes. A 1 KB ASCII response and a 1 KB UTF-8-encoded Chinese response will report very different sizes despite the wire bytes being identical.
- **Fix:** Use `response.headers.get('content-length')` if present, or compute `new TextEncoder().encode(responseText).length` for the actual byte count.

### 19. README claims "SSRF Protection" — not applicable
- **File:** [README.md:53](README.md#L53)
- **Category:** Documentation accuracy
- **What:** README lists "SSRF Protection" under Security Features. Since `fetch()` runs in the user's browser and goes directly to the target URL (no server in between), SSRF — which by definition is server-side request forgery — does not apply to this app.
- **Fix:** Remove that bullet, or rephrase if the intent was something else (e.g. "blocks `file://` / `data:` URL schemes" — which is true and worth keeping under Input Validation).

### 22. `OPTIONS` allowed but body always stripped (and `HEAD` body silently passed earlier)
